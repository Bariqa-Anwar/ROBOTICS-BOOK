## 2. Core Principles

These are the non-negotiable laws of the project.

### Principle 1: Gemini-Only Authorship

**100% of the book prose, code, diagrams, tables, and captions MUST be generated by Gemini 2.5 Flash via the official gemini CLI.** No other model (including Grok, Claude, GPT, Llama, etc.) may contribute even a single sentence.

*Rationale: The project's primary goal is to test and showcase the capabilities of a specific AI model operating via a stateless CLI. Any external contribution invalidates this core experiment.*

### Principle 2: Spec-Driven Development

**Nothing is written or committed unless explicitly required by an approved spec file.** All work must trace back to a specific, version-controlled requirement.

*Rationale: This ensures a rigorous, auditable, and repeatable workflow, which is critical for demonstrating that the AI can follow a plan to produce a complex artifact without continuous interactive guidance.*

### Principle 3: Full Prompt Transparency

**Every single prompt used is saved in `history/prompts/` with YAML frontmatter and committed.** The history must be complete and allow for the reproduction of any part of the book.

*Rationale: Transparency is essential for verifying the "Gemini-Only Authorship" principle and for allowing others to study the human-AI interaction patterns that led to the final product.*

### Principle 4: Embodiment First

**The book itself proves that a stateless CLI model with zero memory can produce coherent long-form technical content when guided by strong specs.** The process is the product.

*Rationale: This principle reminds us that the primary artifact is not just the book, but the demonstration of the entire spec-driven, CLI-based authoring process.*

## 3. Key Standards

- **Exclusive AI Model**: Gemini 2.5 Flash (via gemini CLI) for all content and code generation.
- **Framework**: Docusaurus 3.x + MDX + built-in blog/theme features.
- **Spec Format**: Full Spec-Kit Plus compliance.
- **Writing Style**: Authoritative yet accessible; Flesch-Kincaid Grade 10–12.
- **Tone**: Optimistic but grounded, technically rigorous, forward-looking.
- **Citation Style**: APA 7th edition (embedded links + references page).
- **Diagrams**: Mermaid for architecture/flow; Gemini-generated SVG for other illustrations.
- **Length Target**: 30,000–50,000 words (≈180–280 published pages).
- **Chapters**: 12–15 major chapters + foreword + appendices.

## 4. Constraints

- **Tooling**: Only the `gemini` CLI, `git`, `npm`/`yarn`, Docusaurus CLI, standard Unix/PowerShell tools, and a text editor for specs/prompts are permitted.
- **AI Purity**: No other LLM may touch any book content, even for "polishing" or grammar fixes.
- **Traceability**: Every chapter file must have a corresponding prompt file in `history/prompts/` proving Gemini 2.5 Flash wrote it.
- **Performance**: The final deployed site must score ≥ 97 on all Lighthouse categories.

## 5. Success Criteria

- **Publication**: The book is live and publicly accessible at `https://<username>.github.io/humanoid-robotics` (or the chosen repository name).
- **Authorship Verification**: The book is verifiably 100% written by Gemini 2.5 Flash.
- **Reproducibility**: The complete prompt history is publicly committed and allows for external verification.
- **Quality**: External technical readers rate the book's coherence and accuracy ≥ 8.5/10, despite knowing its origin.
- **Originality**: The book passes plagiarism checks, with all external sources properly cited and rephrased.
- **Build Integrity**: The repository can be cloned and run locally (`npm install && npm run start`) without errors.
- **Legacy**: The project serves as a reference example for AI-driven, long-form technical authoring.

## 6. Accountability

- **Roles**: The human role is strictly defined as Spec Owner, Prompt Engineer, and Final Reviewer. The human is never a direct prose author.
- **Commits**: Every merge to the `main` branch must reference the exact spec and prompt file(s) that justify the change.
- **Integrity Protocol**: Any accidental use of another model immediately invalidates the experiment and the offending commits must be reverted.
